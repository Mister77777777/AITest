## 1. Project Scope / 项目范围

### 1.1 Background / 背景

AutoTestDesign is an AI-driven tool that automates the generation of test artifacts from software requirements. It targets practitioners who follow ISTQB Foundation Level principles and ISO/IEC/IEEE 29119-4 techniques but find manual test-case authoring slow and inconsistent. The tool ingests requirements via CSV, plain text, or direct entry, applies rule-based risk scoring enriched by an LLM-generated rationale, generates black-box test cases through three classical techniques (Equivalence Partitioning, Boundary Value Analysis, Decision Tables), models state behavior to produce white-box (state-transition) cases, synthesizes expected results with algorithmic and LLM-based oracles, and exports the resulting suite as JSON / CSV / Excel.

AutoTestDesign 是一款 AI 驱动的测试用例自动生成工具。面向遵循 ISTQB Foundation Level 原则与 ISO/IEC/IEEE 29119-4 技术、但苦于手写测试用例效率低的实践者。工具接受 CSV、纯文本、手动输入等多种需求来源,先做规则式风险评分并由 LLM 生成理由,然后用三种黑盒技术(EP/BVA/DT)加白盒状态转换技术生成用例,再用算法 + LLM 合成预期结果,最后导出为 JSON/CSV/Excel 供测试管理工具使用。

### 1.2 Objectives / 总体目标

1. Ensure the tool correctly implements all seven Functional Requirements (FR1–FR7) defined in Assignment 2.
2. Verify that test cases generated by the tool satisfy the formal definitions of each technique in ISO/IEC/IEEE 29119-4.
3. Validate that the non-functional requirements (performance, usability, security, maintainability) meet acceptance criteria.
4. Confirm provider portability — the LLM layer works with OpenAI, DeepSeek, Anthropic-compatible proxies and falls back gracefully when no LLM is available.

1. 工具须完整实现 Assignment 2 的 FR1–FR7 七项功能需求。
2. 所生成用例须满足 ISO/IEC/IEEE 29119-4 对每种技术的形式化定义。
3. 非功能需求(性能、易用性、安全、可维护性)达到约定验收标准。
4. LLM 层具备 provider 可移植性,支持 OpenAI、DeepSeek、Anthropic 兼容代理,并能在无 LLM 场景优雅降级。

## 3. High-Level Test Suite Design / 高阶测试套件设计

### 3.1 Mapping Requirements to Techniques / 需求-技术映射

Test technique selection is driven by (a) the nature of the FR and (b) the risk score from R-01 to R-14 in the Risk Analysis Report.

| FR | Primary risk addressed | Selected technique(s) | Rationale |
|----|------------------------|-----------------------|-----------|
| FR1 — Parsing | R-10 (injection), R-11 (encoding) | EP + BVA on file size, encoding, delimiter | Input-boundary driven |
| FR2 — Risk scoring | R-07 (inconsistent score) | EP + DT (keyword presence × category) | Categorical rules → DT natural |
| FR3 — Black-box gen | R-05 (incorrect output) | BVA (for algorithm itself), property-based | Mathematical correctness |
| FR4 — State machine | R-06 (DSL parse fail) | STT (on a reference state machine) + EP on DSL variants | Structural + input |
| FR5 — Oracle | R-01 (LLM malformed) | DT (out-of-range × llm-available matrix) | Four algorithmic paths |
| FR6 — Export | R-13 (TMS mismatch) | EP (JSON / CSV / xlsx × empty / 1-case / many-case suite) | Format-path matrix |
| FR7 — Optimizer | R-05 (missed cases) | DT (duplicate × priority × max_cases) | Multi-dim rule table |
| LLM client | R-02 / R-03 (network, provider) | STT on retry state machine + EP on responses | Stateful retry + input |
| UI | Manual | Exploratory + user-task walkthrough | Non-deterministic interaction |

FR3 的"三种黑盒技术"本身是被测对象,测试它们需要再来一层"元"测试 — 用 pytest 手写期望值做断言(see Section 3.3).

### 3.2 Test Suite Breakdown / 测试套件分解

**TS-1 Core Algorithm Suite / 核心算法套件** — 13 pytest modules under `tests/unit/`, validating C1–C12 against hand-computed expected outputs. Fast (< 1 s), runs on every commit.

**TS-2 Integration Suite / 集成套件** — `tests/integration/test_pipeline.py`, wires all FRs together with `FakeLLM` stub. Validates that the cross-module contracts hold.

**TS-3 LLM-Mocked Suite / LLM 模拟套件** — embedded in unit suites via `tests/fixtures/fake_llm.py`. Ensures LLM-dependent paths don't regress on schema shape.

**TS-4 Real-LLM Smoke Suite / 真实 LLM 冒烟套件** — `scripts/smoke_test.py`. Run before each release / demo; requires valid `.env/config.json`. Validates provider compatibility and end-to-end UX.

**TS-5 UI Manual Suite / UI 手工套件** — documented in Section 5; executed by a reviewer following the golden-path + 5 exploratory scenarios.

**TS-6 Non-Functional Suite / 非功能套件** — performance timing, security grep, provider swap test. Run before each release.

**TS-7 Detailed Module Suite (sample) / 抽样深度套件** — a selected major module gets white + black box combined testing for grading and demonstration (see the Detailed Test Design document for Boundary Value Analysis chosen as the sample).

### 3.3 Technique Choice Per Suite / 每个套件的技术选择

| Suite | Techniques applied | Justification |
|-------|---------------------|---------------|
| TS-1  | EP + BVA + DT + STT | Matches the generators under test; each FR gets the technique it claims to produce |
| TS-2  | STT (pipeline stages as states) | End-to-end flow is a state-transition system |
| TS-3  | DT on (retry × error-class) | Small rule table exhaustively coverable |
| TS-4  | Exploratory + checklist | Real provider behaviour too varied for scripted oracles |
| TS-5  | Use-case + exploratory | User journeys are story-driven |
| TS-6  | EP (file size / response time buckets) | Clear threshold boundaries |
| TS-7  | BVA + STT | Combined black + white for grading depth |

## 6. Chosen Testing Framework and Rationale / 测试框架与理由

### 6.1 Framework Stack / 框架栈

| Purpose | Framework | Version |
|---------|-----------|---------|
| Unit / integration test runner | `pytest` | ≥ 8.0 |
| Coverage measurement | `pytest-cov` | ≥ 5.0 |
| Property / parametrized cases | `pytest.mark.parametrize` (built-in) | n/a |
| Mocking external LLM | Custom `FakeLLM` fixture (no extra dep) | — |
| Static lint | `ruff` | ≥ 0.5 |
| UI smoke | `streamlit run` + `curl` HTTP probe | n/a |
| Real-LLM smoke | `scripts/smoke_test.py` (plain Python) | — |

### 6.2 Rationale / 选型理由

**pytest** — the de-facto standard for Python; native assert rewriting produces readable diffs; `parametrize` supports data-driven tests which fit BVA (6 boundary values per range) and DT (2^N rows) naturally; minimal configuration required.

pytest 是 Python 生态的事实标准;原生 assert 重写让失败信息可读;`parametrize` 天然适配 BVA(一个区间 6 个边界值)与 DT(2^N 行)的数据驱动写法;配置代价极低。

**pytest-cov** — produces line + branch coverage; configurable via `pyproject.toml`; integrates cleanly with the `--cov-report=term-missing` diagnostic used in Task 18 verification.

pytest-cov 提供行/分支覆盖率;`pyproject.toml` 配置;`--cov-report=term-missing` 直接暴露未覆盖行,便于补测。

**FakeLLM (custom fixture)** — preferred over `unittest.mock.patch` because it enforces the interface contract (`structured_call(prompt_name, variables, schema)`) rather than patching arbitrary attributes. This made it trivial to add the pipeline-level integration test with deterministic responses.

自研 FakeLLM 比 `unittest.mock.patch` 更稳 — 它强制走 `structured_call` 这一条路径,不会因 patch 随手替换破坏契约;也正因如此,集成测试能用预排的响应稳定回放。

**ruff** — fast linter that replaces `flake8` + several `pylint` checks; aligns with the pydantic v2 code style used in the project; 100× faster than the combined pre-ruff alternative, keeping pre-commit cheap.

ruff 是新一代 lint,速度快且覆盖 flake8 + 部分 pylint 规则;与项目 pydantic v2 风格一致;pre-commit 里用它开销可忽略。

**Streamlit smoke test** — real end-to-end check that the UI process starts correctly. The `curl -o /dev/null -w '%{http_code}'` probe reliably distinguishes "page renders" from "silent ImportError".

Streamlit 冒烟测试 — 用 `curl` 拿 HTTP 200 作为成功标志,能可靠区分「页面正常」与「悄悄 ImportError」。

### 6.3 Why Not Alternatives / 为什么不选别的

- `unittest` — verbose setup; less expressive than pytest; no built-in parametrize.
- `hypothesis` (property-based) — attractive for BVA but the expected outputs for EP/DT are algebraic enumerations, not properties, so property-based yields little added value for the investment.
- `nose2` — dormant project; pytest ecosystem is deeper.
- `playwright` / `selenium` for Streamlit — overkill; Streamlit's own page objects aren't designed for deep scripting, and the 4-tab UI is trivially manually verifiable.

- unittest 太冗长,缺 parametrize;hypothesis 对 EP/DT 收益低(期望值是代数枚举,不是属性);nose2 社区已停滞;playwright/selenium 对 Streamlit 过度。
