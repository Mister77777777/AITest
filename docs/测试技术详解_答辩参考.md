## 1. 先导：什么是软件测试

### 1.1 一句话定义

**软件测试 = 运行一个程序，检查它的行为是否符合预期，并找出其中的缺陷。**

打个比方：你造了一台自动售货机（软件），测试就是往里面投币、按按钮，看看它有没有吐出正确的饮料、找零对不对。如果投了 5 块却吐了 10 块的饮料，或者按可乐键出来的是雪碧，那就发现了 bug。

### 1.2 为什么不能靠"写完跑一下看看"

你写的 AuthSystem 有 6 条业务规则。如果每个字段的所有可能取值都试一遍，需要测试多少次？

- 年龄：18 到 120，共 103 个值
- 密码：8 到 32 长度的任意字符串，天文数字
- 用户名：3 到 20 长度的任意字符串，同样是天文数字
- 邮箱：包含 @ 和 . 的任意字符串

**穷举是不可能的。**所以我们需要科学的测试设计技术——用最少的用例覆盖最多的缺陷。

### 1.3 ISTQB 是什么

ISTQB（International Software Testing Qualifications Board）是国际软件测试资质认证委员会，定义了软件测试的国际标准。本项目的所有测试技术都遵循 ISTQB Foundation Level 和 ISO/IEC/IEEE 29119-4 标准。

## 2. 被测对象：AuthSystem 是什么

### 2.1 功能概览

AuthSystem 是一个**纯 Python 实现**的用户注册与登录系统，零外部依赖，所有数据存在内存字典里。它位于 `target_app/auth_system.py`。

### 2.2 四个公开方法

| 方法 | 做什么 | 输入 | 输出 |
|------|--------|------|------|
| `register()` | 注册新用户 | username, password, age, email | 无（成功）/ 抛异常 |
| `login()` | 用户登录 | username, password | 无（成功）/ 抛异常 |
| `logout()` | 用户登出 | username | 无（成功）/ 抛异常 |
| `reset_password()` | 重置密码并解锁 | username, new_password | 无（成功）/ 抛异常 |

### 2.3 六条业务规则（BR-01 到 BR-06）

```
BR-01  年龄必须在 18 到 120 之间
BR-02  密码长度 8-32 字符，且至少包含 1 位数字 + 1 位字母
BR-03  邮箱必须包含 @ 和 .
BR-04  用户名必须唯一，长度 3-20 字符
BR-05  连续登录失败 5 次后账户锁定
BR-06  锁定账户可通过密码重置解锁
```

每条规则对应 `auth_system.py` 中的一个验证方法（如 `validate_age()`、`validate_password()` 等）。如果输入不符合规则，系统会抛出对应的异常类（`ValidationError`、`AccountLockedError` 等）。

### 2.4 状态机：一个用户的四种状态

用户在整个生命周期中会在四种状态间切换：

```
Guest ──register()──▶ Registered ──login()──▶ LoggedIn
                           ▲                      │
                           │                   logout()
                           │                      ▼
                           ◀──────────────── Registered
                           
Registered/LoggedIn ──5次失败──▶ Locked ──reset_password()──▶ Registered
```

这对后面的**状态转换测试**至关重要，记在心中。

## 3. 测试设计技术总览

你的 AutoTestDesign 工具使用了 ISTQB 定义的四种测试设计技术：

| 缩写 | 全称 | 中文 | 一句话解释 |
|------|------|------|-----------|
| **EP** | Equivalence Partitioning | 等价类划分 | 把输入分成若干组，每组只测一个代表值 |
| **BVA** | Boundary Value Analysis | 边界值分析 | 专门测边界附近的值，因为边界最容易出 bug |
| **DT** | Decision Table Testing | 判定表测试 | 穷举所有条件组合，适合复杂业务逻辑 |
| **STT** | State Transition Testing | 状态转换测试 | 穷举所有状态和转换路径，适合有状态流转的系统 |

前三种是**黑盒技术**（不看代码，只看需求），最后一种是**白盒技术**（需要了解系统内部状态结构）。

### 为什么需要多种技术？

单一技术有盲区：
- EP 能找到"类型错误"（比如年龄填了字符串），但**测不到边界附近的错误**（比如 `age >= 18` 写成 `age > 18`）
- BVA 能测边界，但**覆盖不了多条件组合**（比如"密码对但账户锁定了会怎样？"）
- DT 能测组合，但**描述不了状态流转**
- STT 能测状态，但**不关心字段值的合法性**

四种技术互补，形成完整的测试覆盖。

## 4. 等价类划分 EP

### 4.1 核心思想

**把输入的所有可能值分成若干"等价类"，每个等价类中任意一个值的行为应该是一样的。然后每个等价类只选一个代表值来测试。**

为什么叫"等价类"？因为同一个类里的所有值在系统眼中是"等价的"——要么全接受，要么全拒绝。

### 4.2 以你的 AuthSystem 为例

看 `target_app/auth_system.py:68-71`，年龄校验：

```python
@staticmethod
def validate_age(age: int) -> None:
    if not (18 <= age <= 120):
        raise ValidationError(f"年龄必须在 18 至 120 之间,实际值: {age}")
```

年龄可以被分成 3 个等价类：

| 等价类 | 范围 | 代表值 | 预期结果 |
|--------|------|--------|---------|
| 有效年龄 | 18 ≤ age ≤ 120 | 59 | 接受（不抛异常） |
| 太小（低于下界） | age < 18 | 17 | 拒绝（ValidationError） |
| 太大（高于上界） | age > 120 | 121 | 拒绝（ValidationError） |

再看密码 `validate_password`（第 74-81 行）—— 密码校验有三个条件：
1. 长度 8-32
2. 至少 1 位数字
3. 至少 1 位字母

密码可以被分成：

| 等价类 | 示例 | 预期 |
|--------|------|------|
| 合法密码（8-32 字符，含数字+字母） | `"Pass1234"` | 接受 |
| 太短（< 8 字符） | `"Ab1"` | 拒绝 |
| 太长（> 32 字符） | `"x" * 33` | 拒绝 |
| 没有数字 | `"Password"` | 拒绝 |
| 没有字母 | `"12345678"` | 拒绝 |

### 4.3 你的工具怎么实现的

代码位置：`autotestdesign/core/blackbox/equivalence.py`

工具会根据字段类型自动推导等价类：

| 字段类型 | 等价类自动生成规则 |
|---------|-------------------|
| **int 整数** | below_min（最小值-1）、valid（中间值）、above_max（最大值+1） |
| **float 浮点数** | below_min（最小值-0.01）、valid（中间值）、above_max（最大值+0.01） |
| **string 字符串** | empty（空串）、normal（合法值）、too_long（超长串） |
| **enum 枚举** | 每个合法值各一个类 + invalid_enum（非法枚举值） |
| **bool 布尔** | true、false |

对于 REQ-001（年龄 18-120，int 类型），工具自动生成 3 条用例：

```
REQ-001-EP-001: age=17  → 拒绝（below_min 类）
REQ-001-EP-002: age=59  → 接受（valid 类）
REQ-001-EP-003: age=121 → 拒绝（above_max 类）
```

### 4.4 EP 的优缺点

| 优点 | 缺点 |
|------|------|
| 大幅减少用例数量（天文数字 → 几个等价类） | 可能遗漏边界附近的问题 |
| 系统化、可复现，不是"凭感觉选值" | 等价类划分的粒度取决于经验 |
| 适合任何有输入域的场景 | 对没有明确输入域的场景（如性能测试）不适用 |

## 5. 边界值分析 BVA

### 5.1 为什么需要 BVA

程序员最容易在**边界**上犯错。比如：

```python
# 正确写法
if 18 <= age <= 120:

# 常见 bug：少写了等号
if 18 < age <= 120:    # 18 岁的用户会被拒绝！
if 18 <= age < 120:    # 120 岁的用户会被拒绝！
```

如果只用 EP，你的代表值 59 根本测不出这个 bug —— 59 离 18 和 120 都太远了。

**BVA 就是专门测边界附近的值。**

### 5.2 标准 6 值法

对于区间 `[min, max]`，ISTQB 建议测试 6 个边界值：

```
min-1,  min,  min+1,  max-1,  max,  max+1
  ↓       ↓      ↓       ↓      ↓      ↓
 略低于  下界   略高于   略低于  上界   略高于
 下界    本身   下界     上界    本身   上界
```

### 5.3 以你的 AuthSystem 为例

对于年龄校验 `[18, 120]`，BVA 生成 6 个值：

| 边界值 | 位置 | 预期 |
|--------|------|------|
| 17 | min - 1（略低于下界） | 拒绝 |
| 18 | min（下界本身） | **接受** ← 这里最容易写错 |
| 19 | min + 1（略高于下界） | 接受 |
| 119 | max - 1（略低于上界） | 接受 |
| 120 | max（上界本身） | **接受** ← 这里也容易写错 |
| 121 | max + 1（略高于上界） | 拒绝 |

对于密码长度 `[8, 32]`：

| 边界值 | 长度 | 预期 |
|--------|------|------|
| `"x" * 7` | 7（min - 1） | 拒绝 |
| `"x" * 8` | 8（下界） | 看其他条件 |
| `"x" * 9` | 9（min + 1） | 看其他条件 |
| `"x" * 31` | 31（max - 1） | 看其他条件 |
| `"x" * 32` | 32（上界） | 看其他条件 |
| `"x" * 33` | 33（max + 1） | 拒绝 |

### 5.4 你的工具怎么实现的

代码位置：`autotestdesign/core/blackbox/boundary.py`

只有**同时定义了 min 和 max** 的字段才生成 BVA 用例。枚举和布尔类型的字段不适用 BVA。

工具对 REQ-001（年龄）、REQ-002（密码长度）、REQ-004（用户名长度）这三个有范围约束的需求各生成 6 条 BVA 用例。

### 5.5 BVA 和 EP 的关系

> **BVA 不是 EP 的替代，而是补充。** EP 保证每种"类型"的输入都被测到，BVA 保证每个边界点都被测到。

对于年龄字段：
- EP 告诉你：要测"小于 18"、"18-120"、"大于 120"三种情况，各一个值
- BVA 告诉你：除了那三种，还要专门测 17, 18, 19, 119, 120, 121

通常的做法是**两个都做**，你的工具就是在 pipeline 中同时对每条需求执行 EP 和 BVA。

## 6. 判定表 DT

### 6.1 核心思想

**如果业务规则涉及多个条件，穷举所有条件组合，检查每种组合的预期结果。**

DT 特别适合"如果 A 且 B，则 X；如果 A 且非 B，则 Y"这种逻辑。

### 6.2 以你的 AuthSystem 为例

BR-05 说"连续登录失败 5 次后账户锁定"。登录涉及多个条件：

| 条件 | 可能的取值 |
|------|-----------|
| 用户存在吗？ | True / False |
| 账户锁定吗？ | True / False |
| 密码正确吗？ | True / False |
| 失败次数已达上限吗？ | True / False |

3 个条件就是 2^3 = 8 种组合，4 个条件就是 2^4 = 16 种组合。DT 会把它们全部列出来。

### 6.3 判定表示例

对于登录逻辑，简化后的判定表（3 个条件，2^3 = 8 条）：

| # | 用户存在 | 账户锁定 | 密码正确 | 预期结果 |
|---|---------|---------|---------|---------|
| 1 | F | F | F | 用户不存在异常 |
| 2 | F | F | T | 用户不存在异常 |
| 3 | F | T | F | 用户不存在异常 |
| 4 | F | T | T | 用户不存在异常 |
| 5 | T | F | F | 密码错误，失败计数+1 |
| 6 | T | F | T | **登录成功** |
| 7 | T | T | F | 账户锁定异常 |
| 8 | T | T | T | 账户锁定异常 |

注意 #1-#4：不管密码对不对、锁没锁定，只要用户不存在，就是"用户不存在异常"。这说明"用户存在"是前置条件，优先级最高——这正是 DT 能帮你发现的逻辑关系。

### 6.4 你的工具怎么实现的

代码位置：`autotestdesign/core/blackbox/decision_table.py`

```python
# 核心算法就一行：
for combo in itertools.product([False, True], repeat=N):
    # 为每一种 True/False 组合生成一条测试用例
```

工具从需求中提取 conditions（条件列表），然后用 `itertools.product` 穷举所有 True/False 组合。N 个条件 → 2^N 条用例。

### 6.5 DT 的适用场景

| 适用 | 不适用 |
|------|--------|
| 业务规则有明确的 if-then 逻辑 | 输入是连续数值（交给 EP/BVA） |
| 多个条件互相影响 | 只有单一条件 |
| 需要验证所有组合都没遗漏 | 条件太多导致组合爆炸（> 6 个条件就要考虑缩减） |

## 7. 状态转换测试 STT

### 7.1 核心思想

**如果系统有明确的状态（比如用户的注册/登录/锁定），画出所有状态和转换，然后测试每条转换路径。**

这是**唯一一种白盒技术**（需要知道内部状态结构），其他三种都是黑盒（不看代码）。

### 7.2 你的 AuthSystem 的状态机

回顾第 2.4 节的状态图：

```
        register()        login()         logout()
Guest ──────────▶ Registered ──────▶ LoggedIn ──────▶ Registered
                      ▲                                  │
                      │    5次连续失败                      │
                      │        ↓                          │
                      ◄─── Locked ◄───────────────────────┘
                         reset_password()
```

用数学方式表达就是：
- 状态集合：{Guest, Registered, LoggedIn, Locked}
- 起始状态：Guest
- 转换集合：
  - Guest → register → Registered
  - Registered → login → LoggedIn
  - LoggedIn → logout → Registered
  - Registered → 5次失败 → Locked
  - LoggedIn → 5次失败 → Locked
  - Locked → reset_password → Registered

### 7.3 两种覆盖准则

你的工具实现了 ISTQB 定义的两种 STT 覆盖准则：

**All-States Coverage（全状态覆盖）：** 确保每个状态都被至少访问一次。
```
Guest → register → Registered                    (覆盖 Guest + Registered)
Registered → login → LoggedIn                     (覆盖 LoggedIn)
Registered → 5次失败 → Locked                     (覆盖 Locked)
```
→ 至少需要 3 条测试用例

**All-Transitions Coverage（全转换覆盖）：** 确保每条转换箭头都被至少走过一次。
```
Guest → register → Registered                    (覆盖转换 1)
Registered → login → LoggedIn                     (覆盖转换 2)
LoggedIn → logout → Registered                    (覆盖转换 3)
Registered → 5次失败 → Locked                     (覆盖转换 4)
Locked → reset_password → Registered              (覆盖转换 5)
LoggedIn → 5次失败 → Locked                       (覆盖转换 6)
```
→ 至少需要 6 条测试用例

### 7.4 你的工具怎么实现的

代码位置：`autotestdesign/core/whitebox/state_machine.py`

核心数据结构：
```python
class Transition:
    source: str    # 从哪个状态出发
    event: str     # 触发什么事件
    target: str    # 到达哪个状态
```

DSL 解析：状态机用类似这样的文本描述：
```
Guest --register--> Registered
Registered --login--> LoggedIn
```

然后工具用 **networkx** 库的 `shortest_path` 算法（Dijkstra）自动计算从起始状态到目标状态的最短路径，生成测试用例。

### 7.5 STT 为什么是"白盒"

黑盒技术不需要知道代码内部结构，但 STT 必须知道系统有哪些状态、状态之间怎么切换 —— 这需要看代码或设计文档。所以它被归类为白盒技术。

## 8. 风险驱动测试

### 8.1 为什么需要风险分析

你的测试资源（时间、人力）是有限的。不可能把所有可能的用例都测一遍。**风险分析帮你决定先测什么、重点测什么。**

想象你是 QA 经理，AuthSystem 有 6 条需求。以下两条你觉得哪个更危险？

- "年龄必须在 18-120"—— 出 bug 最多有人注册不了
- "连续 5 次失败锁定"—— 出 bug 可能导致暴力破解攻击，所有用户数据泄露

显然是第二个。**风险驱动 = 把有限资源投入到后果最严重、最可能出错的地方。**

### 8.2 风险评分公式

你的工具使用标准的 **Likelihood × Impact** 模型：

```
RiskScore = Likelihood × Impact

Likelihood（可能性）: 1-5 分，这个缺陷有多大概率出现？
Impact（影响度）:     1-5 分，一旦出现会造成多大损失？
Score:               1-25 分，综合风险值

Priority 映射:
  得分 ≥ 15  → High   （高风险，必须重点测试）
  得分 8-14  → Medium （中风险，常规测试）
  得分 < 8   → Low    （低风险，简单测试即可）
```

### 8.3 你的工具怎么打分的

代码位置：`autotestdesign/core/risk.py`

**两层打分机制：**

第一层（规则基线 —— 纯算法，不依赖 AI）：
- 检查需求文本中是否包含**高风险关键词**，如：auth、login、security、password、token、加密、认证、权限
- 命中关键词 → impact 从基线 2 提升至 4
- 同样检查**高可能性关键词**，如：concurrent、real-time、network、并发、实时、外部接口
- 非功能需求额外提升 impact

第二层（LLM 增强 —— AI 辅助）：
- 调用大模型（如 GPT-4）生成一段**中文理由**，解释为什么这个风险评分合适
- 如果 LLM 不可用，降级为"规则基线评分"

### 8.4 你的 AuthSystem 的风险分析结果

| 需求 | 风险等级 | 原因 |
|------|---------|------|
| REQ-001 年龄校验 | Low | 边界明确，影响有限 |
| REQ-002 密码规则 | High | 涉及安全核心，含密码关键词 |
| REQ-003 邮箱格式 | Low | 边界明确，影响有限 |
| REQ-004 用户名唯一 | Medium | 涉及数据完整性 |
| REQ-005 登录锁定 | **High** | 涉及安全防护，防暴力破解 |
| REQ-006 密码重置 | **High** | 涉及账户恢复，安全关键路径 |

高风险的需求在后续的**测试套件优化**中会被优先保留，用例也会标记为 High 优先级。

## 9. 测试预言 Oracle

### 9.1 什么是 Oracle

**Oracle（测试预言）= 判断测试结果是"通过"还是"失败"的标准。**

没有 Oracle 的测试毫无意义 —— 你运行了一个用例，但不知道结果是正确的还是错误的，等于白测。

Oracle 的核心问题是：**对于给定的输入，我凭什么说输出是正确的？**

### 9.2 你的工具如何合成 Oracle

代码位置：`autotestdesign/core/oracle.py`

工具使用**三层降级策略**：

```
第1层：越界自动兜底
  ↓ 输入超出了字段的 min/max?
  ↓ Yes → 自动生成："系统应拒绝该输入"（不需要 AI）
  ↓ No  → 进入第2层

第2层：LLM 合成
  ↓ 有大模型可用？
  ↓ Yes → 调 LLM，生成语义化的预期结果描述
  ↓ No  → 进入第3层

第3层：默认模板
  → 根据需求的 expected_actions 生成模板文本
  → "系统应执行以下动作: ..."
```

举例 —— 对于用例 `REQ-001-BVA-001`（age=17）：

- 第1层命中：17 < 18（越界）
- 直接返回：`"系统应拒绝该输入: 年龄 17 低于最小值 18"`
- 没有调用 LLM，节能且准确

对于用例 `REQ-005-DT-005`（用户名存在、未锁定、密码错误）：

- 第1层未命中（所有输入都在合法范围）
- 第2层：调用 LLM，传入需求原文 + 输入值
- LLM 返回：`"系统应拒绝登录并抛出 InvalidCredentialsError"`

### 9.3 设计哲学

**能确定的事情自己判断（算法），不确定的事情才问 AI。** 这样即使 AI 不可用（离线、API 故障），工具仍能完整运行。

## 10. 测试套件优化

### 10.1 为什么需要优化

你的 pipeline 对 6 条需求运行 EP + BVA + DT + STT，可能生成 80+ 条用例。但：

1. **有些用例是重复的** —— 比如 EP 的 valid 类代表值可能和 BVA 的边界值重叠
2. **不是所有用例同等重要** —— 高风险需求应该比低风险需求分配更多测试资源
3. **时间有限** —— 如果只有时间跑 30 条用例，应该跑哪 30 条？

### 10.2 两阶段优化策略

代码位置：`autotestdesign/core/optimizer.py`

**阶段 1：去重**

以 `(需求ID, 测试技术, 输入值)` 作为指纹。完全相同的用例只保留优先级最高的那个。

```
EP-002: age=59 (Priority: Low)
BVA-005: age=59 (Priority: Low)
→ 指纹相同，去重后只保留一条
```

**阶段 2：加权集合覆盖**

问题：80+ 条用例去重后还有 60 条，但我只能跑 30 条，选哪些？

答案：
1. **先保证覆盖**：确保每条需求至少有一条用例被选中（不能让任何一个需求完全没有测试）
2. **再按重要性补充**：剩余名额按优先级权重（High=5, Medium=3, Low=1）从高到低填充

这样即使只跑 50% 的用例，也能确保每条需求都被覆盖到。

## 11. 完整流水线：从需求到用例

### 11.1 Pipeline 全景图

代码位置：`autotestdesign/core/pipeline.py`

```
输入: requirements.csv (6条需求)
  │
  ▼
[Step 1] 需求结构化 (FR1)
  把 "年龄必须在18到120之间" 这种自然语言
  → 解析为: Field(name="age", type=int, min=18, max=120)
  │
  ▼
[Step 2] 风险打分 (FR2)
  用关键词匹配 + LLM rationale
  → 每条需求得到 RiskScore (likelihood, impact, priority)
  │
  ▼
[Step 3] 黑盒用例生成 (FR3)
  对每条需求执行三种技术:
  - generate_equivalence_cases()   → EP 用例
  - generate_boundary_cases()      → BVA 用例
  - generate_decision_table_cases() → DT 用例
  │
  ▼
[Step 4] 状态机用例生成 (FR4)
  - generate_all_states_cases()     → STT 全状态覆盖
  - generate_all_transitions_cases() → STT 全转换覆盖
  │
  ▼
[Step 5] Oracle 合成 (FR5)
  为每条用例生成 expected_result（三层降级）
  │
  ▼
[Step 6] 套件优化 (FR7)
  去重 + 加权集合覆盖
  │
  ▼
[Step 7] 导出 (FR6)
  → JSON / CSV / Excel
  │
  ▼
输出: 82条测试用例的完整套件
```

### 11.2 数据如何流转：一个具体例子

以 REQ-001（年龄 18-120）为例，跟踪一条需求的全流程：

```
1. 输入 CSV 行:

2. 解析 (FR1):

     id="REQ-001",
     raw_text="用户注册时,年龄必须在 18 至 120 岁之间。",

     conditions=["用户在注册页面"],

   )

3. 风险打分 (FR2):
   RiskScore(likelihood=2, impact=2, score=4, priority="Low")

4. 用例生成 (FR3):
   EP:  3条 (below/valid/above)
   BVA: 6条 (17,18,19,119,120,121)

   共13条

5. Oracle (FR5):
   REQ-001-EP-001 (age=17):
     → 第1层越界兜底: "系统应拒绝该输入: 年龄 17 低于最小值 18"
   REQ-001-EP-002 (age=59):
     → 第3层默认模板: "系统应执行以下动作: 系统校验年龄范围"

6. 优化 (FR7):
   13条中去重后剩11条, 因 max_cases 未设限制, 全部保留
   优先级均为 Low, 排在 High/Medium 用例之后

7. 导出 (FR6):
   → test_suite.json / test_suite.csv / test_suite.xlsx
```

## 12. LLM 在测试中的角色

### 12.1 LLM 不是替代测试，而是辅助测试

很多人误以为"AI 测试 = 让 AI 帮你写测试用例"。这不是本项目的方法。

本项目的设计哲学是：

> **确定性算法做主体工作，LLM 做语义增强。LLM 不可用时工具仍可完整运行。**

### 12.2 LLM 在三个环节被使用

| 环节 | Prompt 模板 | LLM 做了什么 | 无 LLM 时的降级 |
|------|------------|-------------|----------------|
| FR1 需求结构化 | `structure_requirement.md` | 从自然语言中提取字段名、类型、范围 | 跳过，需求保持原样 |
| FR2 风险理由 | `score_risk.md` | 生成 1-2 句中文理由解释风险评分 | 返回"规则基线评分（LLM 不可用，已降级）" |
| FR5 Oracle | `synthesize_oracle.md` | 为非越界输入生成语义化预期结果 | 使用模板文本替代 |

### 12.3 三个 Prompt 模板

位置：`autotestdesign/llm/prompts/`

- **`structure_requirement.md`**：指示 LLM 识别字段名、类型（int/float/string/enum/bool）、数值范围（min/max）、枚举值（allowed）
- **`score_risk.md`**：指示 LLM 作为 QA 风险分析师，用 1-2 句中文说明评分合理性
- **`synthesize_oracle.md`**：指示 LLM 根据需求原文、预期动作和具体输入，生成一句中文预期结果

所有 prompt 都要求 LLM 返回严格的 JSON 格式（不允许 markdown 代码围栏）。

### 12.4 为什么 LLM 能辅助测试

测试用例生成有两个子任务：
1. **确定性的**：给定 `age in [18, 120]`，边界值是 17, 18, 19, 119, 120, 121 → **数学公式就能算出来**
2. **语义性的**："用户无法登录时系统应如何提示？" → **需要理解自然语言**

类型 1 用纯算法（EP/BVA/DT 的生成规则），类型 2 用 LLM（rationale、oracle 的自然语言合成）。这就是"规则兜底 + LLM 增强"的混合模式。

## 13. 答辩常见问题预演

### Q1: 你用了哪些测试技术？为什么选这些？

**A:** 四种 ISTQB 标准技术 —— EP（等价类划分）、BVA（边界值分析）、DT（判定表）、STT（状态转换测试）。

选择原因：
- AuthSystem 每条需求都有明确的输入域 → EP + BVA 天然适用
- 登录/注册涉及多条件判断 → DT 穷举组合
- AuthSystem 有明确的状态机（Guest → Registered → LoggedIn → Locked）→ STT 覆盖状态转换

### Q2: 82 条用例是怎么算出来的？

**A:** Pipeline 对 6 条需求分别运行 EP、BVA、DT，再加上状态机的 STT：
- EP：每字段 2-4 个等价类 × 6 条需求 ≈ 18 条
- BVA：每范围字段 6 个边界值 × 3 条有范围的需求 ≈ 30 条
- DT：N 个条件 × 2^N 组合 × 6 条需求 ≈ 24 条
- STT：All-States + All-Transitions ≈ 10 条
- 合计 ≈ 82 条（去重后约 60+）

### Q3: 你怎么验证生成的用例是正确的？

**A:** 两层验证：
1. **自动化**：pytest 测试套件（14 个测试文件，core/ 覆盖率 ≥ 95%），每个模块都有独立的单元测试
2. **交互式审查**：Streamlit UI 支持人工对生成的用例进行编辑、增删、优先级调整

### Q4: LLM 在这里面到底做了什么？

**A:** LLM 不直接生成测试用例，而是做三件"语义增强"的事：
1. 把自然语言需求结构化（提取字段类型和范围）
2. 为风险评分生成人类可读的理由
3. 为测试用例合成自然语言的预期结果描述

所有核心测试逻辑（EP/BVA/DT/STT 的生成规则）都是纯算法，不依赖 LLM。

### Q5: 你的工作和手工测试相比有什么优势？

**A:**
- **效率**：报告中的成本估算显示使用 AutoTestDesign 约 4.3 人日 vs 手工 12.0 人日，节省约 64%
- **一致性**：算法保证每次对同样的需求生成同样的用例，不会因人而异
- **覆盖度**：穷举边界值和条件组合，人工容易遗漏
- **可追溯**：每条用例都标记了来源需求、使用的技术、覆盖的等价类/边界/转换

### Q6: 状态机是怎么描述的？

**A:** 使用自定义 DSL（领域特定语言）：
```
Guest --register--> Registered
Registered --login--> LoggedIn
LoggedIn --logout--> Registered
Registered --failedAttempt--> Registered
LoggedIn --failedAttempt--> LoggedIn
Registered --fifthFailedAttempt--> Locked
LoggedIn --fifthFailedAttempt--> Locked
Locked --resetPassword--> Registered
```

工具解析这个 DSL，构建有向图（networkx），然后用最短路径算法自动生成测试路径。

### Q7: 如果有新的需求加入，整个流程怎么变化？

**A:** 只需要：
1. 在 `requirements.csv` 中添加一行
2. 如果涉及新的状态转换，更新状态机 DSL
3. 重新运行 pipeline → 自动生成新的用例套件

不需要修改任何算法代码。

### Q8: 什么是 coverage（覆盖率）？你的工具怎么衡量？

**A:** 工具中的 coverage 不是代码覆盖率（行覆盖），而是**需求覆盖**：
- `requirement_coverage` = 被至少一条用例覆盖的需求数 / 总需求数
- 优化器保证这个比例达到 100%（每条需求至少有一条用例）
- 另外还记录 **EP 等价类覆盖**、**BVA 边界值覆盖**、**STT 状态/转换覆盖**

代码层面，pytest 配合 `pytest-cov` 可以提供代码行覆盖率。
